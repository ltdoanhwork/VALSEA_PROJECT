    # ... existing config ...

## Development Workflow

### 1. Local Development (No Docker)

```bash
# Terminal 1: Main backend
cd backend
source .venv/bin/activate
uvicorn main:app --reload --port 8001

# Terminal 2: Scorer service
cd mlops/scorer
source .venv/bin/activate
uvicorn app:app --reload --port 8002

# Terminal 3: Frontend
cd frontend
npm run dev
```

### 2. Docker Development

```bash
# Build and run all services
docker compose up --build

# Or just scorer
docker compose up scorer --build

# View logs
docker compose logs -f scorer
```

### 3. Training Workflow

```bash
cd mlops/scorer

# 1. Generate training data
python generate_data.py

# 2. Train model
python train.py

# 3. View experiments
mlflow ui --port 5000

# 4. Evaluate
python evaluate.py

# 5. Test locally
uvicorn app:app --reload --port 8002

# 6. Rebuild Docker with new model
docker compose up scorer --build
```

## Best Practices

### Data
- вњ… Version your training data (add timestamps, git tags)
- вњ… Keep raw data separate from processed data
- вњ… Never commit large datasets to git (use DVC or S3)

### Models
- вњ… Never commit model binaries to git (too large)
- вњ… Use model registry (MLflow) for tracking versions
- вњ… Save model metadata (training date, accuracy, dataset version)

### Code
- вњ… Use type hints (`def score(text: str) -> float`)
- вњ… Add docstrings to all functions
- вњ… Write tests before training (test data loading first)

### MLflow
- вњ… Log everything: params, metrics, artifacts
- вњ… Use meaningful experiment names
- вњ… Tag runs with metadata (e.g., `git_commit`, `dataset_version`)

### Docker
- вњ… Use multi-stage builds to keep images small
- вњ… Don't copy `.venv/` or `mlruns/` into container
- вњ… Mount models as volumes (easier updates without rebuild)
